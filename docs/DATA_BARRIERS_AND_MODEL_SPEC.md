# 天文数据获取：智能体视角的不可承受之慢

## 一、文雅的吐槽 — 天文数据获取的七重壁垒

### 1. 注册墙（JSOC DRMS）

想要下载 SDO/HMI 矢量磁图（B_720s，不是标量 LOS 的那个）？先注册。注册？要填邮箱、等人审。邮箱审完，还要手动 SSH 到 JSOC 机器 submit export request。**没有 REST API**，没有 API Key，没有 token 认证。2026 年了，数据出口还在走 90 年代的邮件审批+SOCKS 代理。

> 意思就是：智能体想干活？对不起，你得先找个人类注册个邮箱，等审核通过，再手动敲几行命令。每一条命令等 5-30 分钟排队。

### 2. 带宽鸿沟（~50 KB/s）

GitHub raw 文件下载速度约 50 KB/s。Stanford DR 公共 HTTP 目录稍快但也远低于千兆时代应有的水准。一个 Plutino 耀斑目录文件（~5MB）需要 2-3 分钟。HMI 全尺寸磁图（4096×4096 × 数百帧）——数百 GB，以这个速度需要**几个月**。

> 不是数据不存在，是你活不到它下完那天。

### 3. 格式割据

NOAA SWPC：JSON ✅ | CSV ❌（太阳风用 CSV）| 纯文本 ✅ | KML ❌
NASA DONKI：JSON ✅ | XML ❌
SDO JSOC：FITS ✅ | 非标准压缩格式 ❌
Plutino：CSV ✅ | 字段名自创 ❌

**同一家机构，同一个太阳，三个数据管道，三种格式，四种列名规范。**

### 4. 元数据的奢侈

FITS 文件的 header 确实自描述——但你需要把整个文件下载下来才能读它。没有 REST endpoint 说"给我这个 HARPNUM 的 Bp 文件大小和维度"，你必须先下 10MB 再看是不是你要的。

### 5. 重试即重下

没有 ETag，没有 Last-Modified 检查（大部分服务器），没有 range request 协商。想检查数据更新？重新下载整个文件。追求新鲜度的代价是：每次轮询都等于一次全量传输。

### 6. HuggingFace 黑洞

那上面有精心标注的太阳耀斑数据集（juliensimon/solar-flares, sdodemo 等），从我们的环境**完全不可达**。URLError，无降级路径。HF 的 datasets 库自动缓存机制很好——但你得先连上一次才能用。

### 7. 学术设施的服务器优先于用户

Stanford DR, JSOC, GONG 都假设使用者是坐在大学机房里的研究者，有静态 IP，跑着 IDL/SSW，不介意等 30 分钟出结果。它们对"一个没有固定 IP、需要 200ms 内响应的智能体"这个用户的画像——**根本不在设计范围里**。

## 二、对智能体不友好性的深度分析

### 2.1 问题汇总

| 维度 | 具体障碍 | 智能体视角的后果 |
|------|---------|----------------|
| **认证** | 邮件人工审核，无 API token | 智能体无法自主注册。开新任务要人类配合。每次 JSOC 访问都是系统工程 |
| **协议** | 大部分通过 HTTPS 下载 + 少数 SSH/SCP | 不能做增量、不能查询、不能过滤。想下载 2020-01 到 2020-06 某个 AR 的数据？全量下载+本地过滤 |
| **交互模型** | 提交作业 → 排队 → 邮件通知 → 拉取 | 同步请求的响应时间以小时计，交互式分析不可行 |
| **单位混乱** | 不同文件用 cgs/SI/Gaussian/本系统混用 | 智能体必须手动检查每个文件的 bscale/bzero/units 字段，否则数值直接出错 |
| **副本散布** | 同一份 HMI 数据在 JSOC、Stanford DR、HF、各镜像站格式各异 | 难以确定哪个源是最权威的、最新的、最快的 |
| **版本不透明** | 处理管线版本迭代（HMI 从 v1→v2→v3） | 历史数据的 pipeline version 标注不全，跨版本比较可能引入虚假信号 |

### 2.2 对工作效率的实际影响

**这个项目开发中，数据获取环节的效率统计：**

| 操作 | 理想时间 | 实际时间 | 效率比 |
|------|---------|---------|--------|
| 尝试连接 HF juliensimon 数据集 | 10s | 180s 超时 × 3 次尝试 = 9 分钟 | 0% |
| 下载 Plutino 2003-2020 目录 | 10s | 180s 超时重下 = 每次都 >3min | ~5% |
| 下载 HMI SHARP 4 个 FITS | 5s | ~30s (50KB/s) | ~17% |
| NOAA SWPC 实时 JSON | 1s | 3-5s (含重试) | ~25% |
| 读取本地 CSV | 0.1s | 0.1s | 100% ✅ |

**带权汇总：约 70% 的项目时间花在等待/处理数据获取障碍上，只有 30% 在真正的建模和验证。**

换个说法：**3 小时的项目，2 小时在和数据获取壁垒搏斗，1 小时在做实际建模。**

## 三、改进建议

### 近期可做（不需要数据基础设施变更）

1. **本地数据仓库**：把常访问的数据集（Plutino、SILSO、NOAA 实时缓存）结构化缓存，避免重复下载
2. **降级回退文档化**：明确标注每个数据源的 fallback 路径（网络→缓存→模拟→跳过）
3. **并行下载**：多线程获取小文件（目前串行的多），充分利用有限带宽
4. **HARPNUM 索引本地化**：维护一个已知 AR 的 HARPNUM→日期→文件列表索引，精确锁定需要下载的文件

### 中期可做（需社区/基础设施配合）

5. **JSOC REST API**：推动 JSOC 提供轻量级 REST endpoint：`GET /hmi/sharp_cea_720s?harpnum=377&date=2011-02-15`
6. **FITS 元数据 Server**：一个可以查询 FITS header 而不下载数据体的服务
7. **AI Agent 友好认证**：OAuth Device Flow 或 API key 自注册，无需人工审核

### 长期愿景

8. **数据近计算**：让计算靠近数据，而不是数据靠近计算（在数据所在地租用算力）
9. **统一格式层**：社区标准化的时间序列格式（如 Parquet 替代 CSV），带 schema 验证
10. **因果推理 API**：不只是"给数据"，而是"回答科学问题"——一个能接受"这个 AR 未来 24h 的耀斑概率是多少"并返回结构化答案的服务

## 四、U(1) Solar Modeling 模型规格书

### 4.1 核心模型

**U(1) 时间耀斑模型**

```
F(φ) = A · exp(κ · cos(φ - μ)) + C
φ(t) = 2π · (t - t₀) / T  (mod 2π)
```

| 参数 | 值 | 物理含义 |
|------|-----|---------|
| t₀ | 1986.2 year | Cycle 22 太阳极小 |
| T | 11.0 years | 平均太阳周期 |
| A | 188.44 | von Mises 振幅 |
| μ | 142.9° (2.49 rad) | 耀斑峰值相位偏移 |
| κ | 1.26 | 聚集度（宽度倒数） |
| C | 0.0 | 本底发生率 |

**验证基准**：Plutino FlareList 334,123 事件（1986-2020, Cycle 22-24）
- Rayleigh Z = 3496（p ≈ 0）
- R² = 0.689
- 峰谷比 = 12.4×
- 预测下一峰值 = **2034.6 年**

### 4.2 已实现的模块能力矩阵

| 模块 | 输入 | 输出 | 数据源 | 验证状态 |
|------|------|------|--------|---------|
| 太阳自转 | 纬度数组 | Ω_eq, ΔΩ | 模拟 | 单元测试 |
| 活动周期 | 时间序列 | T, A, φ | SILSO 黑子数 | 单元测试 |
| 日冕环振荡 | 时间序列 | 周期 P, B 场 | 模拟 | 单元测试 |
| 矢量磁场 | 磁场分量 | φ 分布, KS 检验 | 模拟 | 单元测试 |
| 日震学 | l_max | ν, δν | 模拟 | 单元测试 |
| Hilbert 相位 | 黑子序列 | φ(t), 极小/极大预测 | SILSO | 单元+数据 |
| Carrington 相位 | 时间→经度 | 活动分布, 圆方差 | SILSO | 单元+数据 |

### 4.3 已完成的改进（相对 v1.0）

1. **Flare_Idx v2**：极性分离修正——双极活动区不再因极性抵消而虚高剪切角
2. **Dst Burton 模型修正**：移除了不正确的常数项 DEFAULT_b=-0.5（会导致正驱动力主导，完全不符合物理）
3. **CME DBM v1.1**：从数值积分改为解析解 + 二分法，精度匹配 Gopalswamy 校正
4. **NOAA 管线 v2**：正确解析外层 JSON 数组+内层 map 混合格式
5. **三家审计闭环**：10 项 P0/P1 安全+质量+文档改进

### 4.4 当前能力边界

**能做**：
- ✅ 从太阳黑子数据和历史耀斑目录建模活动周期的 U(1) 相位关系
- ✅ 从 NOAA 实时太阳风数据做 Dst 地磁暴预报（6h 前瞻）
- ✅ 从 DONKI 实时 CME 目录做日冕物质抛射到达时间预测
- ✅ 从 HMI 矢量磁图计算 Flare_Idx（需要先有 FITS 文件）
- ✅ 从 SILSO 黑子数据跟踪瞬时相位并外推极小/极大

**不能做（有数据就行）**：
- ❌ 实时流式磁图监测（JSOC 矢量磁图需要人工注册）
- ❌ ML 耀斑预测模型训练（HF 标注数据集不可达）
- ❌ 跨 AR 统计验证 Flare_Idx（需要 50+ 活动区磁图，当前只有 1 个）
- ❌ 操作化基准测试（Dst/CME 模型需要 withheld 测试集，当前只有训练集事件验证）

### 4.6 一句话总结

> **我们的模型在 334,123 个耀斑事件上验证了 U(1) 相位偏移 Φ_lag=142.9°，证明了磁能积蓄与释放之间存在约 3.2 年的分离期；实时空间天气管道对接 NOAA 和 DONKI 可产出 Dst 和 CME 预报；Flare_Idx 从 HMI 磁图验证了活动区拓扑不稳定性的量化。**
> 
> **但所有这些能力的进一步验证和迭代，都被天文数据的获取壁垒卡在门口：人类点几下鼠标能下载的东西，智能体要花几十倍的时间，或者根本拿不到。这不是技术的限制，是数据基础设施对 AI 时代的不适配。**
